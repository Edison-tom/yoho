# Yoho（呦吼）桌面宠物 App · 完整产品方案 V3.1

> 版本：V3.1（优化版） | 更新日期：2026-07-04（优化版）
> 一句话定位：一颗悬浮在桌角的种子，一只偶尔动一动的宠物，和一个只有悬停时才出现的名字——它们安静地陪着你，直到达成目标的那一天。


## 一、产品哲学与核心定位

### 1.1 解决什么问题？
* 孤独奋斗：考研、备考、写论文或加班时，缺乏即时的正面反馈与陪伴感。
* 目标稀释：长期目标（如考研、攒钱）难以量化，容易因枯燥而放弃。
* 异地联结：情侣间缺少一种“不查岗、不打扰”却又能感知彼此存在的轻量方式。

### 1.2 核心价值
* 把“努力”翻译成“生命感”：专注 30 分钟 -> 宠物零食 -> 宠物排泄（肥料） -> 浇灌种子 -> 树成长 -> 结果。
* 不打扰的温柔：窗口极小（240×320），默认半透明，无主动弹窗，一切操作由用户悬停或点击触发。
* 双生模式：覆盖“单身独行”与“情侣共赴”两种人生状态；情侣模式下拥有专属宠物 + 一只可在双方电脑间串门的共养宠物。


### 1.3 宠物品种（不可变）

> **总则**：Yoho 提供 4 种默认宠物品种 — 2 猫 2 狗，终生不可更改。
> 这个决定源于创始人对自家宠物的感情：银渐层和布偶是家里的两只猫。
> 狗选了中国保有量最高的两个品种：泰迪和金毛。

| 品种 | 类型 | 英文标识 | 性格 | 选定理由 |
|:---|:---|:---|:---|:---|
| 🐱 银渐层 | 猫 | `silverShaded` | 温顺、安静、亲人 | 创始人家养，圆脸绿眼银灰渐变 |
| 🐱 布偶 | 猫 | `ragdoll` | 温柔、黏人、慵懒 | 创始人家养，蓝眼长毛重点色 |
| 🐶 泰迪 | 狗 | `poodle` | 活泼、聪明、亲人 | 中国保有量最高的犬种 |
| 🐶 金毛 | 狗 | `goldenRetriever` | 忠诚、温柔、阳光 | 中国保有量第二的犬种 |

用户首次启动时从以上 4 种中任选 1 只，选定后**品种不可更换**（可在设置中修改宠物名字）。
这是产品的情感锚点：你的宠物是独一无二的伙伴，不是可以随意替换的皮肤。


## 二、核心玩法闭环（驱动力引擎）

这是整个 App 的“黄金闭环”，每一步都提供即时多巴胺反馈：

专注奋斗 30分钟 → 获得 1块 爱心饼干 → 投喂宠物 → 宠物排泄产出 1颗 肥料 → 拖拽肥料至树根 → 树成长 / 阶段提升 → 是否到达 100% 目标肥料数？ → 是 → 🍇 截止前一日结果达成

### 2.1 奋斗计时（自动无感）
* 判断依据：系统未休眠 + 键盘/鼠标空闲 < 5分钟（双保险）。
* 计量单位：每 30 分钟 有效专注 = 1 块“爱心饼干”。
* 防卷保护：每日软上限 8 块（对应4小时高效专注），超出不累计。
* 新手礼包：首次安装赠送 1 块新手饼干 + 1 颗新手肥料，安装后 5 分钟内即可体验"投喂→排泄→施肥→种子生长"的完整闭环。
* 首日不设上限：安装当天不受每日 8 块软上限限制，让新用户充分感受玩法循环的乐趣。

### 2.2 喂养与排泄（趣味转化）
* 投喂：用户长按并拖拽“零食堆”到宠物嘴边，宠物呈现咀嚼动画。
* 排泄物（肥料）：投喂后约 3-5 分钟，宠物随机排出一颗可爱化肥料（默认为“金色星星”或“巧克力球”）。
* 堆积机制：桌面最多同时存在 5 颗肥料，若溢出，宠物会扒拉屏幕边缘提醒用户“该施肥啦”。

### 2.3 施肥与成长（时间倒推 + 80% 宽松规则）

> 不设固定肥料数。树的成长曲线由目标日期反向推演，按"宽松平均值"计算。

* 操作：拖拽肥料到树根区域，触发"浇灌"光效。
* 计算逻辑（核心公式）：
  1. 用户设定目标截止日期（如：2028 年 1 月 1 日）。
  2. 系统自动计算剩余天数 D = 目标日期 − 今天。
  3. 按"宽松平均每日专注"估算肥料基数：默认每人每天 2 小时有效专注 = 4 块饼干 = 4 颗肥料。
  4. 目标肥料总数 = D × 4。
  5. 6 个阶段按百分比均匀分布。
  6. 🌱→🍇 按每天 2 小时专注的正常节奏，树恰好于**截止日期前一日结果**。
  7. "80% 宽松规则"：即使平均每天只有 1.6 小时（80% 的节奏），进度稍慢，但截止前仍可结果。
* 举例：目标"2028-01-01 前攒够首付 5 万买车"
  - 假设今天为 2026-07-04，剩余 D ≈ 546 天。
  - 目标肥料总数 = 546 × 4 = **2184 颗**（每天 2 小时专注 = 4 颗）。
  - 按正常节奏，到 2027-12-31（截止前一日）刚好积累 2184 颗 → 🍇 结果。
  - 即使平均每天只专注 1.6 小时（80% 节奏），进度稍慢，仍可在截止前结果。
  - 如果某天专注了 4 小时（双倍节奏），进度加快，结果会提前到来。

| 阶段 | 进度（占目标肥料总数的 %） | 视觉形态 | 触发文案（树冠气泡展开时显示） |
| :--- | :--- | :--- | :--- |
| 🌰 种子期 | 0% | 花盆右下角被"抠出"一个弧形横截面，可见盆内土壤中埋着一颗种子；土壤微微起伏，种子间歇发光（随肥料累积频率递增） | "梦想已种下，等你的第一份肥料" |
| 🌱 萌芽期 | 20% | 破土嫩芽，两片叶子 | "发芽了！你的努力让它冒了头" |
| 🌿 成长期 | 40% | 小树苗，分出枝桠 | "它站稳了，继续喂它吧" |
| 🌳 繁茂期 | 60% | 茂盛树冠，出现花苞 | "枝繁叶茂，离收获只差一场雨" |
| 🌸 开花期 | 80% | 满树花朵，花瓣飘落 + 全屏烟花 | "花开满枝，果实已经在路上了！" |
| 🍇 结果期 | 100% | 挂满发光果实 + 金色光晕 + 全屏庆祝 | "你的坚持结出了果实，每一分努力都算数！" |
## 三、界面与交互设计（极简主义）

### 3.1 窗口规范
* 尺寸：默认 240px × 320px（可缩放至最小 200×280）。
* 层级：常驻所有窗口最上层，但不抢占系统焦点。
* 透明度：默认 80% 半透明，鼠标悬停时恢复 100% 不透明，移出后 3 秒渐隐。
* 位置：默认吸附屏幕右下角，支持自由拖拽移动。
* 多显示器：默认仅在主显示器显示；设置中可开启"跟随当前活动屏幕"，多屏用户在任一屏幕工作时窗口自动跟随。

### 3.3 树冠悬浮标签（树名的呈现）
* 默认态：树冠上方仅有一个直径 6px 的呼吸光点（极致简洁）。
* 悬停态：鼠标悬停树冠区域，光点放大为磨砂气泡，显示完整树名 + 当前阶段图标（如“📚 北大上岸树 · 🌿”）。
* 触发态：里程碑达成时，气泡短暂放大 3 秒显示庆祝语，随后自动收缩。

### 3.5 社交与分享（零成本病毒传播）
* 截图分享：任意窗口状态下，右键菜单 → "截取当前画面"，自动生成带 Yoho 水印的精美分享图，可直接复制到剪贴板或保存为 PNG 文件。
* 结果期纪念海报：每次目标达成（结果期）时，自动生成一张纪念海报，**支持一键分享到微信**（含树名、阶段图标、目标日期、总专注时长、总肥料数、宠物形象）。
* 海报规格：1080×1920 竖版，适配微信朋友圈和聊天窗口，底部带 Yoho 品牌标识。
* 免费用户可保存并分享；Pro 用户的海报无水印，支持自定义文案、选择主题色，可生成多种尺寸。


## 四、多模式深度解析

### 4.0 多关系并行：一个人可以同时拥有多棵树

> 你有男朋友，也有闺蜜——Yoho 不会让你二选一。

#### 4.0.1 核心规则
* 每个用户默认拥有一棵**个人树**（单身模式）。
* 用户可以同时加入：
  - **最多 1 段情侣关系**（对应 1 棵情侣树）
  - **最多 N 个队伍**（每个队伍对应 1 棵队伍树，不限制同类型队伍数——你可以同时有"大学闺蜜队"和"公司闺蜜队"两棵闺蜜树）
* 你的宠物只有 1 只，陪伴你穿梭于所有树木之间。饼干是你专注的成果，你可以选择投喂给任意一棵树。
* 每棵树有自己的肥料数、阶段和进度，互不影响。

#### 4.0.2 界面规则：最多同时显示 3 棵树

> 你有 5 棵树不代表桌面上要同时摆 5 棵。保持清爽，Yoho 替你排优先级。

**默认显示规则**（用户未手动指定时）：
* 窗口内最多同时显示 **3 棵树**（缩小为缩略卡片并列排布）。
* 系统自动选择**距离截止日期最近**的 3 棵树优先显示（时间紧迫的树更值得关注）。
* 个人树不受此限制——如果个人树不在前 3，它仍作为第 1 棵保留，占用一个位置。
* 超出 3 棵的树折叠到「+N 棵」按钮中，点击展开**树详情面板**。

**手动选择**：
* 在树详情面板中，用户可**拖拽排序**或**钉选**任意树木到主窗口。
* 钉选的树始终显示，不再被自动排序淘汰。
* 取消钉选后恢复自动排序。
* 不管手动还是自动，窗口内始终 **≤ 3 棵**。

**单树 / 多树显示差异**：
| 树数量 | 窗口布局 |
|:---|:---|
| 1 棵 | 全窗口展示，树占满中央，宠物在侧 |
| 2-3 棵 | 每棵树缩小为缩略卡片（约 60×80px），横向排列，当前活跃树高亮 |
| ≥4 棵 | 前 3 棵显示 +「+N」按钮在末尾 |

**交互**：
* 点击某棵缩略树 → 该树展开为主视图。
* 双击或点击「+N」→ 打开树详情面板，一览所有树（含截止日期倒计时、当前阶段、成员头像）。
* 宠物始终在屏幕一角，不随树切换改变。

* 个人树仅自己可见；情侣树和队伍树分别对关系内成员可见。

#### 4.0.3 饼干分配
* 饼干是全局的（你的专注成果），不按树拆分。
* 拖拽饼干到哪棵树的区域，就投喂给那棵树。
* 同一块饼干只能投喂一次，不可复制。

#### 4.0.4 场景举例
> 小美有男朋友阿强（情侣关系），和大学室友组了「姐妹暴富」闺蜜队，和公司同事另组了「摸鱼不耽误搞钱」闺蜜队。
> - 她今天专注 2 小时，获得 4 块饼干。
> - 2 块喂情侣树「一起买房树」。
> - 1 块喂「姐妹暴富」闺蜜树。
> - 1 块喂「摸鱼不耽误搞钱」闺蜜树。
> - 阿强和他的兄弟也有一个老铁队「兄弟搞钱树」，把饼干全喂给了老铁树。
> - 小美同时拥有 4 棵树：1 棵个人 + 1 棵情侣 + 2 棵闺蜜。Yoho 不做限制。

### 4.1 首次启动流程
用户首次进入 App：
0. 登录/注册：支持邮箱注册、Google 账号、GitHub 账号三种方式（MVP）；Apple 登录、微信扫码（上线后接入）。Supabase Auth 统一管理。
1. 选宠物（从 4 种默认免费品种中挑选 1 只）：
   - 🐱 银渐层 · 🐱 布偶 · 🐶 泰迪 · 🐶 金毛
2. 设定第一个目标（按 §5.1 流程，默认单身模式种一棵个人树）。
3. 进入主界面，种子种下。
* 模式不在此步骤选择。用户默认为"单身"，拥有一棵个人树。后续可通过「建立恋爱关系」（§4.3）、加入老铁队（§4.5）、加入闺蜜队（§4.6）来扩展关系——**一个人可以同时拥有多段关系、多棵树**（见 §4.0）。

### 4.2 单身模式（个人树）
* 每个用户都有一棵个人树，这是 Yoho 的基础模式。
* 树：完全私有，仅消耗自己产出的肥料。
* 宠物：仅有一只（银渐层或布偶或泰迪或金毛，首次启动时自选），位于界面左侧。
* 互动菜单：单击宠物 -> ①喂食 ②摸摸头（免费，仅提供情绪价值）。
* 昵称：可设置自己的昵称（默认「主人」），宠物在对话气泡和摸摸头互动中使用此称呼。例如宠物会说「主人，来摸摸头吧！」。入口：设置 → "我的昵称"。
* 文案调性：“你一个人，也走了很远的路。”

### 4.3 建立恋爱关系（从单身到情侣）

> 你已经一个人种过树、养过宠、走过一段路。现在，有人想和你一起种一棵新的。

#### 4.3.1 入口与触发
* 入口：设置 → "开启情侣模式"（单身用户可见；已绑定情侣的用户不显示此入口）。
* 前置确认弹窗：

> 🌳 开启情侣模式后，你当前正在种的树「{树名}」将被封存到森林档案中。
> 你和伴侣将一起种一棵全新的树，设定共同的目标。
>
> 你的宠物会留在你身边，成为你的专属宠物。
>
> [我再想想]  [继续]

#### 4.3.2 邀请流程
* 用户点击"继续"后，系统生成 **6 位配对码**，有效期 24 小时。
* 分享方式：复制配对码 + 一段预设邀请文案，通过微信/短信发送给伴侣。
  - 预设文案示例：「我在 Yoho 种了一棵树，想和你一起种一棵新的。这是我的邀请码：XXXXXX，下载 Yoho 后输入就能找到我。」
* 伴侣侧：
  - **已有 Yoho 账号**：在设置 → "输入配对码"中输入，立即绑定。
  - **新用户**：下载 Yoho → 注册 → 首次目标设定时选择"我有配对码" → 输入配对码 → 完成绑定后进入情侣模式首次设定流程。

#### 4.3.3 树的处理
* 发起方的当前树：自动存入森林档案，标记「🌸 因开启恋爱模式封存」，附带封存日期。
* 若伴侣也是已有用户且正在种树：伴侣的树同样存入其森林档案，标记同上。
* 森林档案中该树的卡片展示：树名 + 目标类型 + 达成阶段（封存时的进度）+ 粉色丝带角标。
* 双方共同种一棵新树：重新走 §5.1 目标设定流程（模式固定为情侣，跳过模式选择步骤）。
* 封存的树不计入"未完成"，也不计入连续彩蛋；它是一个独立的中性状态——"不是放弃，是翻篇了"。

#### 4.3.4 宠物的处理
* 发起方：原有宠物自动转为"专属宠物"（品种不变，饼干和肥料清零重新开始）。
* 伴侣侧：
  - 新用户：在首次设定中选择自己的专属宠物（4 种默认免费品种任选，可与发起方重复）。
  - 已有用户：若已有宠物，同样转为自己的专属宠物；若无（刚注册），则首次设定时选择。
* 双方共同选择 1 只共养宠物（4 种默认免费品种任选，可与专属重复）。
* 后续运作完全遵循 §4.4 情侣模式的标准规则。

#### 4.3.5 仪式感设计（轻量）
* 绑定完成后、仪式动画前，双方各自设置昵称（见 §4.4.6）：
  - **我的昵称**：显示给对方看的称呼，默认「主人」。
  - **称呼 Ta**：对方在你界面上显示的名字，默认「宝宝」。
  - 双方各自独立设置，不必对称（你可以叫 Ta「宝宝」，Ta 可以叫你「主人」）。
* 昵称确认后，进入**共同目标设定**：
  - 发起方提出共同目标（树名 + 截止日期 + 目标类型），伴侣侧弹出确认弹窗：
    > 🌳 {callTa} 提议一起种一棵「{treeName}」
    > 目标：{target} | 截止：{date}
    > 目标肥料数：{count} 颗
    > [修改] [确认]
  - 伴侣可修改后发回，双方来回确认直到一致。双方都点「确认」后，共同目标生效。
  - 确认即锁定：共同目标一旦生效，任何一方要修改，需**双方重新确认**（见 §5.6）。
* 共同目标确认后，双方屏幕同时显示：

> 🎀 你们的故事，从今天开始一起写。
>
> 「{用户A昵称}」和「{用户B昵称}」在 Yoho 种下了第一棵共同的树。

* 动画：两片花瓣从屏幕两侧飘入，在树的位置汇合，闪烁一下后消散（约 3 秒，不阻塞操作）。
* 金句匹配：绑定后第一次触发金句时，情侣版金句库立即生效。
* 森林档案中"因恋爱封存"的树自动附带一条专属注记：「这棵树封存于你遇见 Ta 的那天。」
* 单身期间完成的树：不受影响，保持原有状态。若用户在单身期间种过多棵树，森林档案中"因恋爱封存"的树与之前的完成树并列——一个人的路和两个人的路，都在同一片森林里。

#### 4.3.6 特殊场景：双方都是已有单身用户
* 当双方都是已在 Yoho 种过树的单身用户时：
  - 谁的配对码被输入，谁就是"发起方"；但流程对双方是对称的。
  - 两人的当前树**各自**存入森林档案（标记「🌸 因开启恋爱模式封存」）。
  - 两人的原有宠物**各自**转为专属宠物。
  - 其余流程与上述一致。
* 设计意图：不区分"谁主动"——这是一段关系的开始，不是一个人的决定。


### 4.4 情侣模式

情侣模式采用“双专属 + 一共养”的三宠体系，兼顾现实感与仪式感。

#### 4.4.1 专属宠物（每人一只，从 4 种默认免费品种中自选）
* 你和伴侣各自从银渐层、布偶、泰迪、金毛 4 种中选 1 只作为专属宠物，分别位于共享树的左右两侧。
* 各自的饼干可以投喂给自己的专属宠物（产生自己名下的肥料），也可以投喂给对方的专属宠物（见 §4.4.4 伴侣互动菜单），肥料计入共享树。
* 两只专属宠物品种可以相同也可以不同（例如：你选银渐层，伴侣选金毛）。
* 状态显示（反查岗设计）：
  - 界面永不显示“在线/离线”文字。
  - 伴侣活跃时，其专属宠物呈彩色动态；长时间空闲/休眠时，呈睡姿剪影，身边出现闪烁信封。

#### 4.4.2 共养宠物（你们一起养的“孩子”）
* 情侣绑定后，双方共同从 4 种默认免费品种中选择 1 只作为共养宠物，位于共享树的正下方。
* 品种可与专属宠物重复（例如：你的专属是银渐层、伴侣专属是金毛、共养也选银渐层——完全 OK）。
* 共养宠物可以在两台电脑之间“串门”——点击它，它会从当前屏幕边缘“走出去”（约 3 秒俏皮动画），出现在伴侣的屏幕上。
* 串门规则：
  - 共养宠物每次只在一方的屏幕上显示。
  - 当前“拥有”共养宠物的一方，可投喂它（消耗 1 块饼干），产出 **双倍肥料**（2 颗），直接计入共享树。
  - 喂完后共养宠物停留 10–30 分钟，然后自动跑回对方屏幕。
  - 也可手动点击“让它回家”立即送回。
* 串门情绪价值：当你看到共养宠物从伴侣那边跑过来——意味着“Ta 在专注工作，把宠物送来陪你了”。

#### 4.4.3 形象定制（“即将开放”预留入口）
* 入口位置：设置 → 宠物管理 → “上传照片生成卡通形象”（按钮置灰，标注“即将开放”）。
* 功能规划：上传真实宠物照片 → AI 生成 Q 版卡通形象 → 替换游戏内宠物外观。
* 适用范围：专属宠物和共养宠物均可定制。
* 商业化：¥9.9/次（见§8.1），暂不开发，入口先行占位。

#### 4.4.4 伴侣互动菜单（点击对方专属宠物）

| 动作 | 消耗 | 你的操作提示 | 对方反馈 |
| :--- | :--- | :--- | :--- |
| 替 Ta 喂宠物 | 消耗 1块 饼干 | 点击后弹出：「{称呼Ta}，我来替你喂宠物了！」 | 对方宠物跳跃接住 + 咀嚼动画；对方屏幕弹出「{对方给你的昵称} 替你喂了宠物🍪」。共享树获得 1 颗肥料 |
| 送爱心 | 消耗 3颗 肥料 | 点击后弹出：「{称呼Ta}，送你一颗心❤️」 | 全屏飘心，树冠跳动 |
| 送吻 | 消耗 5颗 肥料 | 点击后弹出：「{称呼Ta}，亲一个💋」 | 触发共养宠物跑到对方身边，全屏粉色烟花 |
| 留言信封 | 免费 | 输入文字；对方上线时信封自动拆开，弹出「{对方给你的昵称} 给你留了一段话💌」 | 对方屏幕弹出留言内容 |

#### 4.4.5 离线投喂机制
* 若对方离线，你对 Ta 专属宠物的所有投喂/互动不会丢失，转化为“星光信封”存于对方宠物身边。对方上线点击信封，可一次性接收所有离线期间的思念与投喂。
* 共养宠物在对方离线期间，自动停留在在线一方的屏幕上；对方上线后，可选择是否立即送回。

#### 4.4.6 昵称设置

> 昵称贯穿单身和情侣两种模式。单身时你的宠物用它叫你；恋爱后它成为你们互相称呼的纽带。

* **我的昵称**（单身 & 情侣通用）：
  - 默认值：「主人」。
  - 单身模式：宠物在对话气泡和互动中使用，例如「主人，来摸摸头吧！」。
  - 情侣模式：伴侣界面中的宠物标签、互动提示、绑定仪式文案中显示。
* **称呼 Ta**（情侣模式专属）：
  - 默认值：「宝宝」。
  - 你给伴侣起的称呼，在你界面中用于对方宠物标签、互动菜单、森林档案注记等所有原本文案中「Ta」的位置。
* 入口：
  - 单身模式：设置 → "我的昵称"。
  - 情侣模式：设置 → 伴侣管理 → "编辑昵称"（可同时修改我的昵称和称呼 Ta）；或首次绑定后自动弹出。

* 昵称规则：
  - 长度限制：2-8 个中文字符或 4-16 个英文字符。
  - 单身模式：「我的昵称」可随时修改，即时生效。
  - 情侣模式：双方独立设置，不必对称。修改后通过 SSE 推送同步到对方屏幕。
  - 分手解绑后：你的昵称恢复为单身模式默认值；对方昵称在森林档案中显示为「一位旧友」（见 §6.2）。
* 昵称在以下场景中生效（单身模式仅使用「我的昵称」，情侣模式两项均生效）：

| 场景 | 使用昵称 | 示例（假设"我的昵称"=主人，"称呼 Ta"=宝宝） |

| 替 Ta 喂宠物（互动菜单） | 称呼 Ta | 点击后弹出对话：「宝宝，我来替你喂宠物了！」 |
| 送爱心（互动菜单） | 称呼 Ta | 点击后弹出对话：「宝宝，送你一颗心❤️」 |

| 树冠气泡（情侣 solo 金句） | 称呼 Ta | 「宝宝也在努力，你们在朝同一个方向走。」 |

> 你有对象，也有兄弟——两棵树不冲突（见 §4.0）。
> 几个兄弟，一棵树，一个目标——谁说男人不能一起种树？

#### 4.5.1 模式定位
* 人数：**2-10 人**（最少 2 人激活，上限 10 人）。
* 关系：兄弟、室友、队友、同事——任何"一起扛过事儿"的关系。
* 调性：硬核、热血、不矫情。"话不多，就是干。"
* 默认昵称体系：
  - 「我的昵称」默认值：兄弟
  - 「称呼 Ta」无需设置——老铁模式不搞一对一昵称，统一称呼为「兄弟」。
  - 可自定义昵称（如"老王""阿强"），但不强制。

#### 4.5.2 核心机制
* **共享一棵树**：所有成员的肥料汇入同一棵树。
* **共同目标**：发起人提出目标（树名 + 截止日期 + 目标类型），所有成员需逐一确认。全体确认后生效，任何修改需全员重新确认（见 §5.6）。
* **每人一只专属宠物**：从 4 种默认免费品种中自选，品种可重复。
* **无共养宠物**：兄弟之间不存在"共有的一只"，各自养各自的。
* **组队邀请**：发起人生成 **6 位组队码**，有效期 48 小时（比情侣码宽松，方便协调时间）。其他人输入组队码加入。加入后立即获得队伍树 + 自己的专属宠物。

#### 4.5.3 宠物串门（组队灵魂机制）

> 老铁之间没有共养宠物，但彼此的宠物可以"串门"——这是组队模式最有温度的设计。

**串门触发方式**：
* 成员 A 点击成员 B 的宠物 → 选择「让它来串门」
* 成员 A 的专属宠物从屏幕边缘"哒哒哒走出去"（约 2 秒离开动画）
* 成员 B 屏幕边缘出现 A 的宠物，"哒哒哒走进来"（约 2 秒到达动画）
* 到达后，宠物头顶气泡显示「{sender}的{petBreed}来找你啦！」

**串门期间的交互**：
* 宠物的"主人"变为当前屏幕持有者（临时投喂权转移）
* 持有者可单击串门宠物 →「摸摸头」→ 产生 1 颗肥料计入**串门发起人的**共享树
* 持有者也可投喂串门宠物 → 消耗自己 1 块饼干 → 产生 2 颗肥料（**1 颗归树 + 1 颗作为串门礼物返还给宠物主人**）
* 串门宠物持续停留 10-30 分钟，之后自动跑回原主人屏幕

**串门冷却**：同一只宠物每天最多被邀请串门 3 次，防止滥用。

**串门情绪价值**：当你忙得焦头烂额，兄弟的宠物突然跑过来蹭你——"他在干嘛？哦，在喂我的宠物。"

#### 4.5.4 组内互动

| 动作 | 消耗 | 效果 |
| :--- | :--- | :--- |
| 替兄弟喂宠物 | 消耗 1 块饼干 | 对方宠物跳跃接住，对方屏幕弹出「{sender} 替你的{petBreed}喂了一口」。共享树获得 1 颗肥料 |
| 拍肩膀 | 免费 | 对方宠物抖动一下，对方屏幕弹出「{sender} 拍了拍你」。无肥料产出，纯社交 |
| 喊一嗓子 | 免费 | 全组成员屏幕弹出「{sender}：兄弟们冲！」。组内共用的喇叭消息，每人每天限 3 次 |
| 🎲 随机串门（§4.5.3） | 自动触发 | 系统随机挑选成员的宠物，自动跑到另一成员屏幕上。到达方可以摸摸头/投喂，产出的肥料计入共享树 |

#### 4.5.5 进度与排名
* **无排名**：不显示谁贡献了多少肥料，避免内卷和比较。
* 界面仅显示「已共同获得 {count} 颗肥料」和「剩余 {count} 天」。
* 设计原则："一起扛，不分你我。"

#### 4.5.6 退出与解散
* 成员可主动退出，肥料数不撤回，树继续生长。
* 若人数降至 1 人（只剩 1 人），树自动封存入该成员的森林档案，标记「老铁未完成」。
* 发起人可解散队伍，所有成员的树各自封存。

#### 4.5.7 金句系统
* 复用个人版金句（奋斗/激励向），另配 20 句「老铁专属金句」：
  - 调性：热血、不矫情、兄弟情。
  - 示例：
    - 「兄弟齐心，树都能种成森林。」
    - 「别掉队，都在等你。」
    - 「今晚谁先到 4 块饼干，谁请夜宵。」
    - 「几个大老爷们种树，说出去谁信？」

### 4.6 闺蜜模式

> 你有男朋友，也有闺蜜——两棵树不冲突（见 §4.0）。

## 五、目标设定与轮回系统

> Yoho 有两层目标：**个人小目标**（只有自己知道，悄悄努力）和**共同目标**（你们约定一起完成的事，需互相确认）。

### 5.0 个人小目标（所有模式通用）

* 每个人都可以设置一个**私人小目标**，只有自己能看到。
* 与树的共同目标互不影响——树按时间/肥料自动成长，小目标是你的私房计划。

#### 5.0.1 设置与展示
* 入口：点击自己的专属宠物 → 「我的小目标」。
* 设置内容：
  - 目标描述（必填，如"每天背 50 个单词""读完 5 本书""瘦 3 斤"）
  - 计量方式（可选）：次数/金额/时长/纯文字
  - 目标值（可选，如"50 次""30 天"）
* 展示位置：宠物身边一个便签纸片，悬停展开完整内容。
* 隐私：**不同步、不共享、不上传**。仅本地存储，换设备需手动迁移。

#### 5.0.2 进度更新
* 手动更新：点击便签 → 「+1」或输入数值。
* 完成时宠物弹出：「{myName}，你的小目标达成了！🎉」。
* 可随时修改或放弃，无惩罚。

### 5.1 首次目标设定流程（4 步，约 40 秒）
1. 选目标类型（此时默认为个人树目标；情侣/队伍目标在建立关系时另行设定，见 §5.6）。
2. 设定目标截止日期（必填，树的成长唯一基准）：
   - 核心原则：“时间是最公平的尺度”。无论你的目标是攒钱、减肥还是考试，最终都要落到一个日期上。
   - 输入方式：日历选择器，精确到日（如：2028-01-01）。
   - 系统根据日期自动计算总肥料数（§2.3 公式），树的阶段按百分比分布。
3. 选目标类型 + 辅助计量（帮助感知进度，不影响树的成长速度）：
   - 预设模板：考研/考公、旅行/体验、买车/买房、健身/减肥、技能/创作，或自定义。
   - 辅助计量（可选填写，仅用于界面展示“进度感”）：
     - 💰 金额（如：50,000 元 → 界面显示“已存 32,000 / 50,000”，但树仍按时间×80% 判定）
     - 📝 次数（如：100 次健身 → 界面显示“已完成 67 / 100 次”）
     - 无辅助计量（纯靠时间驱动）也可，界面仅显示“剩余 XXX 天”。
4. 给树命名（强制）：
   - 输入框占位符：“例如：北大上岸树 / 首付买车树”
   - 设计心机：树名默认生成，但鼓励手动修改，专属命名极大增强情感绑定。

### 5.2 轮回系统（截止前一日结果，80% 节奏即够用）
* 达成判定：累计肥料达到目标总数的 **100%**（结果期），即触发“目标达成”。
* 正常节奏（每天 2 小时专注）下，结果恰好发生在截止日期前一日。
* 即使平均每天只有 1.6 小时（80% 节奏），进度稍慢，仍可在截止前结果。
* 达成后自动触发：
  1. 全屏庆祝烟花 + 成就弹窗 + 自动生成可微信分享的海报
  2. 当前树自动存入“森林档案”
  3. 宠物弹出气泡：“接下来，想种一棵什么树？”
  4. 点击后复用首次设定流程（模式沿用，宠物数据保留，肥料重置为 0）

### 5.3 森林档案（回顾系统）
* 入口：主界面左下角图标（🌲），悬停显示“我的森林”
* 展示：卡片流展示所有已达成目标（含树名、目标类型、达成日期、总肥料数、纪念海报）
* 交互：点击卡片可回看树的最终形态（只读模式）
* 连续达成彩蛋：
  - 第1棵：解锁“新手园丁”称号 + 小草帽
  - 第3棵：解锁树皮肤（银杏/樱花）
  - 第5棵：解锁肥料皮肤
  - 第10棵：专属成就勋章 + 金色主题

### 5.4 主动放弃目标（灵活切换）
* 路径：设置 → 我的目标 → “放弃当前目标，重设新目标”
* 后果：当前树存入森林，标注“未完成”，不计入连续彩蛋
* 文案：“这条路暂时不走了，但每一步都算数。种棵新树吧！”

### 5.5 情侣模式下的轮回协同
* 双方都达成：同时触发庆祝，一起进入新目标设定
* 一方想换目标：需双方都达成后才可更换（防止进度不一致）
* 一方主动放弃：树标记“未完成”，双方解绑，各自进入单身模式


## 六、分手处理机制

### 6.1 核心原则
> “尊重过去，释放未来，让告别也体面温柔。”
* 用户主导：所有操作由主动提出解绑的一方发起
* 不强制删除：数据归属权清晰，但绝不自动销毁任何数据
* 无惩罚性：没有任何“扣分”、“惩罚”或“负向反馈”
* 允许复合：解绑后仍可通过原码重新绑定

### 6.2 解绑流程（三步完成）
路径：设置 → 伴侣管理 → 解除绑定

| 步骤 | 交互 | 文案 |
| :--- | :--- | :--- |
| 1. 确认 | 弹窗：“确定要和 [伴侣昵称] 解除绑定吗？” | 按钮：[再想想] / [确定解除] |
| 2. 选择数据归属 | 弹窗显示当前树的状态，让用户选择如何处理 | 见下方“数据归属选项” |
| 3. 完成 | 双方App同步解绑，各自进入单身模式 | 显示告别语 |

### 6.3 数据归属选项
| 选项 | 执行逻辑 | 适用场景 |
| :--- | :--- | :--- |
| 🌲 剪断连理枝（默认） | 当前树立即结果，存入双方森林，标注“曾经共同种过”。各自宠物/零食/肥料完全保留 | 和平分手 |
| 🙈 各自安好 | 当前树立即结果，存入双方森林。森林中伴侣昵称变为“一位旧友”，不显示具体是谁 | 不想再看到对方痕迹 |
| 🔒 封存记忆 | 所有共同数据打包加密，用户设置6位数字密码。封存后数据不出现在主界面/森林，仅通过“记忆胶囊”查看 | 想要珍藏回忆 |
| 🗑️ 就此翻篇 | 当前树不存入森林（视为未完成）。双方宠物重置为初始状态，所有共同数据清零 | 希望彻底重新开始 |

* 注意：无论哪种选项，各自上传的宠物照片、个人设置、个人目标记录永远不会被删除或共享给对方。

### 6.4 告别语
* 主动发起方：“你已解除了和 [伴侣昵称] 的绑定。树已结果，路还很长，愿你继续生长。”
* 被通知方：“[伴侣昵称] 解除了和你的绑定。你种过的树会留在森林里，而你值得一片新的森林。”

### 6.5 复合机制
* 路径：设置 → 伴侣管理 → “重新绑定伴侣”
* 逻辑：输入对方6位码，绑定后恢复情侣模式，森林档案中恢复对方信息
* 彩蛋：复合后的第一棵新树，树根处多一颗金色小星星，悬停显示：“这是你们重逢后种的第一棵树。”


## 七、密码封存记忆（隐私保护）

### 7.1 设计原则
> “不是删除，不是公开，而是体面地封存——只有你愿意时，才能重新打开。”

### 7.2 可封存内容
* 共同种的树（全部阶段记录、施肥明细、里程碑文案）
* 聊天/留言记录（所有互动留言、时间戳）
* 宠物互动记录（投喂次数、送爱心/吻的时间线）
* 共同目标详情（目标类型、截止日期/金额）
* 纪念海报（树达成时自动生成的海报图）

### 7.3 封存流程（分手时）
在数据归属选项中选择“🔒 封存记忆”：
1. 弹窗提示：“这段回忆值得被珍藏。设置一个只有你知道的密码，未来随时可以打开。”
2. 输入6位数字密码（两次确认）
3. 设置密码提示问题（可选）：“你们第一次旅行的城市是？”
4. 完成封存，对方App显示：“[伴侣昵称] 封存了这段回忆。”

### 7.4 “记忆胶囊”入口（查看封存内容）
* 位置：设置 → 底部“记忆胶囊”区域
* 显示：封存条数（如“1段封存的记忆”）
* 交互：
  1. 点击 → 弹窗：“输入密码即可打开这段封存的回忆”
  2. 输入正确 → 展示封存内容（只读模式，可浏览所有数据）
  3. 密码错误 → 提示重试（不限制次数）
  4. 忘记密码 → 显示密码提示问题；若仍无法回忆，可选择“永久删除此段记忆”（需二次确认）

### 7.5 双方封存状态独立
* 对方封存 → 你显示“[伴侣昵称] 封存了这段回忆”
* 对方删除 → 你显示“[伴侣昵称] 选择不保留这段回忆”
* 对方长期离线 → 30天后自动按“封存记忆”处理

## 九、“少打扰”原则（防弃用设计）

* 无声音：默认 0 音效（可设置内开启）
* 无弹窗：所有提示以气泡/角标/动画形式在窗口内呈现，绝不弹出系统级对话框
* 无被动推送：不主动提示“该奋斗了”、“宠物饿了”（宠物不会饿死，肥料最多堆积5颗）
* 专注隐身模式：一键完全透明，适合开会/录屏/投影场景
* 数据不焦虑：不显示“距离目标还差 XX 天/元”，仅显示“当前阶段”，弱化具体缺口数字

### 10.2 可视化事件时间线
* 入口：设置 → "查看运行日志"。
* 以聊天记录形式展示事件流水，按颜色区分：
  - 🟢 绿色：正常事件（获得饼干、施肥成功、宠物串门）
  - 🟡 黄色：警告（同步重试、离线队列积压、内存偏高）
  - 🔴 红色：错误（同步失败、窗口崩溃恢复、数据校验失败）
* 支持按类型筛选：仅看错误 / 仅看同步 / 仅看宠物。
* 每条事件可点击展开，显示技术细节（面向开发者，用户无需关心）。

### 10.3 用户友好的异常提示
* 当程序内部出错时，**不弹出英文技术对话框**。改为：

> ⚠️ Yoho 遇到了一点小问题
> 不用担心，你的数据没有丢失。
> 问题已自动记录。如果持续出现，点击"导出诊断报告"发给开发团队。
>
> [导出报告]  [一键上报]  [忽略]

* 最多每 24 小时弹一次，避免打扰。

### 10.4 一键上报（静默收集）
* 入口：上述弹窗中的"一键上报"按钮，或设置 → "发送诊断数据"。
* 点击后自动收集并上传：
  - 诊断报告（同 10.1）
  - 程序错误堆栈（技术层面，用户不可见）
  - 最近 200 条内部日志（JSON 格式，含时间戳和调用链）
* 上传方式：通过 HTTPS 发送到服务端诊断接口，无需用户填任何信息。
* 上报完成后提示：

> ✅ 诊断数据已安全发送
> 本次仅上传错误诊断信息，不包含你的任何个人数据。
> 感谢帮助 Yoho 变得更好！

* 安全与隐私：
  - 上报数据**仅包含程序运行状态和错误日志**，不含用户输入的密码、宠物名称、留言内容等私人信息。
  - 所有上报数据经 AES 加密传输。
  - 服务端数据保留 90 天，仅用于 Bug 修复分析，到期自动清除。
  - 不上报：用户地理位置、IP 之外的设备指纹、文件系统路径。


## 十一、技术实现要点（MVP 版本）

| 模块 | 方案 | 说明 |
| :--- | :--- | :--- |
| 跨端框架 | **Tauri（Rust + 前端）** | 确定采用 Tauri，打包体积<10MB，内存占用低，原生性能远优于 Electron |
| 自动计时 | CGEventSource（Mac）/ GetLastInputInfo（Win） | 检测键盘/鼠标空闲 + 系统休眠状态 |
| 数据同步 | **SSE + HTTP REST，离线优先操作日志** | 比 WebSocket 更简单可靠，天然支持断线重连；离线期操作存入本地队列，上线后批量同步 |
| 本地存储 | SQLite | 存储宠物数据、本地缓存、离线操作队列、记忆胶囊 |
| 宠物/树渲染 | CSS 关键帧动画 + Lottie；复杂骨骼动画使用 **Spine 2D** | 避免高功耗 3D，Spine 可提供更流畅的宠物连续动作（跑动、跳跃） |
| 窗口属性 | Mac: .floating / Win: WS_EX_TOPMOST | 置顶且无边框、透明背景 |
| 加密存储 | AES-256，密钥由用户密码经 PBKDF2 派生 | 服务端零知识架构——服务器无法解密用户封存的记忆 |
| 国际化 | 代码层使用 `i18next` 预留接口，MVP 仅支持中文 | 避免后期大规模重构，所有字符串统一管理 |
| 用户认证 | **Supabase Auth**，邮箱 / Google / GitHub（MVP 三种）；Apple / 微信（上线后接入） |微信扫码登录（后续接入，需备案域名） | 所有登录方式共用一个 user_id，数据完全互通 |
| 自动更新 | **Tauri updater plugin**，启动时检查服务端版本 JSON | 发现新版本 → 静默下载 → 用户确认后重启安装；更新失败自动回滚到旧版本 |
### 11.1 服务器方案

> 核心原则：Yoho 的专注计时完全在本地完成，服务器仅负责情侣间数据中转和云端存储。单身模式用户几乎不需要服务器。

#### 11.1.1 服务器职责与负载评估

| 功能 | 负载 | 说明 |
| :--- | :--- | :--- |
| 用户注册/登录 | 极低 | 仅在首次使用和换设备时触发 |
| 情侣配对（6 位码） | 极低 | 一次性操作，生成绑定关系 |
| 实时数据同步（SSE） | **中等** | 每 30 分钟专注才触发一次状态变更，实际请求频率很低 |
| 离线消息中转 | 低 | 存储"星光信封"，对方上线后一次性推送 |
| 记忆胶囊 Pro（云端存储） | 低 | 存加密 JSON 数据，体积极小（纯文本为主） |
| 诊断数据上报 | 极低 | 用户手动触发，日均请求量可忽略 |
| 纪念海报存储 | 低 | PNG 图片，每张数十 KB |

### 11.2 服务器环境配置

#### 11.2.1 操作系统
* 推荐 Ubuntu 22.04 LTS（64 位）或 CentOS Stream 9。
* 内核版本 ≥ 5.15，支持 Tauri 打包所需的系统调用。

#### 11.2.2 后端技术栈
* 语言：Rust（与 Tauri 客户端共享代码）或 Node.js（若需快速迭代）。
* HTTP 框架：Axum（Rust）/ Express（Node.js）。
* SSE 推送：基于 HTTP 长连接，无需额外中间件。
* 数据库驱动：sqlx（Rust）/ pg（Node.js）。

#### 11.2.3 数据库与中间件

| 组件 | 方案 | 说明 |
| :--- | :--- | :--- |
| 关系数据库 | PostgreSQL 13+ | 用户表、配对关系、同步日志、胶囊索引 |
| 缓存（可选） | Redis | 仅在线用户状态缓存，前期可不用，直接查数据库 |
| 消息队列（可选） | — | 不需要。离线消息直接写数据库，对方上线时 SQL 查询即可 |
| 反向代理 | Nginx | HTTPS 终结 + 静态资源 + 限流 |

#### 11.2.4 数据库核心表（MVP 最小集）

```sql
-- 用户表
users (id, username, password_hash, created_at)

-- 情侣绑定
couples (id, user_a, user_b, pair_code, bound_at)

-- 同步事件日志（离线优先队列）
sync_events (id, couple_id, from_user, event_type, payload, created_at)

-- 记忆胶囊
memory_capsules (id, user_id, couple_id, encrypted_data, locked_at)

-- 诊断上报
diagnostic_reports (id, user_id, report_data, uploaded_at)
```

#### 11.2.5 Nginx 基本配置要点
* HTTPS 强制（Let's Encrypt 免费证书，certbot 自动续期）。
* SSE 长连接需要关闭 proxy_buffering：`proxy_buffering off;`
* API 路由 `/api/*` 反向代理到后端端口。
* 静态资源（海报）直接走 OSS CDN，不经过服务器。
* 限流：单 IP 60 次/分钟，防止恶意攻击。
* 版本更新接口：`GET /api/version/latest` → 返回 `{ "version": "1.2.0", "url": "https://...", "notes": "..." }`

#### 11.2.6 部署与运维

| 事项 | 方案 |
| :--- | :--- |
| 部署方式 | Docker Compose 一键部署（含 API + PostgreSQL + Nginx） |
| CI/CD | GitHub Actions，推送 main 分支自动构建 Rust 二进制并 rsync 到服务器 |
| 日志 | 应用日志输出到 stdout，由 systemd journal 收集，无需额外日志服务 |
| 监控 | 初期用 `htop` + `docker stats` 手动巡检；成长期接入阿里云云监控（免费版） |
| 备份 | PostgreSQL `pg_dump` 定时任务（每日凌晨 3 点），备份文件同步到 OSS |
| 告警 | 服务宕机时通过阿里云短信服务发送通知（¥0.045/条，几乎无成本） |
